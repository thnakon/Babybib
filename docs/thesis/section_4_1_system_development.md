# บทที่ 4 ผลการดำเนินงาน

## 4.1 ผลการพัฒนาระบบ

ผลการพัฒนาระบบสร้างบรรณานุกรม APA 7th Edition อัตโนมัติ (Babybib) ประกอบด้วยรายละเอียดด้านสถาปัตยกรรมระบบ การออกแบบฐานข้อมูล เครื่องมือและเทคโนโลยีที่ใช้ในการพัฒนา โครงสร้างไฟล์ระบบ รวมถึงมาตรการรักษาความปลอดภัย ดังรายละเอียดต่อไปนี้

---

### 4.1.1 สถาปัตยกรรมระบบ (System Architecture)

ระบบ Babybib ได้รับการออกแบบและพัฒนาโดยใช้สถาปัตยกรรมแบบ **Server-Side Rendering (SSR)** ผ่านภาษา PHP ในการประมวลผลฝั่งเซิร์ฟเวอร์ ร่วมกับ JavaScript สำหรับการประมวลผลแบบ Real-time ในฝั่งไคลเอนต์ เพื่อให้ระบบมีความยืดหยุ่น สามารถรองรับผู้ใช้งานได้หลากหลายกลุ่ม ทั้งผู้ใช้ทั่วไป (Guest) สมาชิก (Member) และผู้ดูแลระบบ (Administrator) โดยสถาปัตยกรรมระบบแบ่งออกเป็น 4 ชั้น (Layers) ดังนี้

**1) ชั้นการนำเสนอ (Presentation Layer)**

ชั้นการนำเสนอเป็นส่วนที่ผู้ใช้งานติดต่อกับระบบโดยตรง ออกแบบด้วยภาษา HTML5 จัดรูปแบบด้วย CSS3 โดยใช้แนวคิด CSS Custom Properties (Variables) เพื่อให้สามารถควบคุมธีมสีและการออกแบบจากจุดเดียว มีการประยุกต์ใช้เทคนิค Glassmorphism สำหรับส่วนของผู้ใช้ทั่วไป เพื่อให้หน้าตาของระบบมีความทันสมัยและเป็นมืออาชีพ สำหรับส่วนของผู้ดูแลระบบ (Admin Panel) ใช้ TailwindCSS ร่วมกับ Alpine.js เพื่อสร้างส่วนต่อประสานผู้ใช้ที่มีประสิทธิภาพ รองรับ Dark Mode และมีไอคอนจาก Lucide Icons ส่วนระบบฟอนต์ใช้ Google Fonts (Inter สำหรับภาษาอังกฤษ และ Tahoma สำหรับภาษาไทย) ไอคอนใช้ Font Awesome 6.4 ทั้งนี้ทุกหน้าจอออกแบบตามหลัก Responsive Web Design เพื่อรองรับการแสดงผลบนอุปกรณ์ทุกขนาดหน้าจอ ตั้งแต่โทรศัพท์มือถือ แท็บเล็ต จนถึงคอมพิวเตอร์ตั้งโต๊ะ

**2) ชั้นตรรกะทางธุรกิจ (Business Logic Layer)**

ชั้นนี้รับผิดชอบการประมวลผลหน้าที่หลักของระบบ พัฒนาด้วยภาษา PHP 8.2 ตามหลัก Modular Design โดยแยกไฟล์ตามหน้าที่การทำงานอย่างชัดเจน ประกอบด้วยโมดูลหลักดังต่อไปนี้

- **โมดูลการจัดรูปแบบบรรณานุกรม (APA 7th Formatter):** ทำหน้าที่แปลงข้อมูลดิบที่ผู้ใช้กรอกเข้ามาให้เป็นรูปแบบบรรณานุกรมตามมาตรฐาน APA 7th Edition โดยอัตโนมัติ รองรับทรัพยากร 10 หมวดหมู่ ครอบคลุม 38 ประเภทย่อย สามารถสร้างทั้งรายการบรรณานุกรม (Bibliography) การอ้างอิงแบบ Parenthetical และการอ้างอิงแบบ Narrative ได้ในครั้งเดียว
- **โมดูลค้นหาอัจฉริยะ (Smart Search v3.1):** ระบบค้นหาอัตโนมัติที่สามารถตรวจจับประเภทข้อมูลนำเข้า (ISBN, DOI, URL หรือ Keyword) ได้โดยอัตโนมัติ รวมถึงตรวจจับภาษาไทย/อังกฤษ เพื่อเลือกแหล่งค้นหาที่เหมาะสม โมดูลนี้ใช้ cURL Multi-handle ในการเรียก API จากหลายแหล่งพร้อมกัน (Parallel HTTP Requests) เพื่อลดเวลาในการรอผลลัพธ์
- **โมดูลการจัดเรียง (Sorting Module):** รับผิดชอบการจัดเรียงรายการบรรณานุกรมตามมาตรฐาน APA 7th Edition โดยจัดเรียงภาษาไทยก่อนภาษาอังกฤษ จัดเรียงตามชื่อผู้แต่ง จัดเรียงตามปี และเพิ่มลำดับอักษรต่อท้ายปี (year suffix) เมื่อผู้แต่งและปีซ้ำกัน เช่น ก, ข, ค สำหรับภาษาไทย หรือ a, b, c สำหรับภาษาอังกฤษ
- **โมดูลการจัดการสมาชิก (User Management):** รับผิดชอบระบบสมัครสมาชิก การล็อกอิน/ล็อกเอาท์ การยืนยันอีเมลด้วยรหัส OTP 6 หลัก การรีเซ็ตรหัสผ่านผ่านลิงก์อีเมล การจัดการโปรไฟล์ และการอัปโหลดรูปประจำตัว
- **โมดูลการจัดการโครงการ (Project Management):** ให้ผู้ใช้สร้างโครงการ (Project) สำหรับจัดกลุ่มรายการบรรณานุกรม สามารถย้ายรายการระหว่างโครงการ และส่งออกบรรณานุกรมทั้งโครงการในรูปแบบ Microsoft Word (.DOCX) หรือ PDF ได้
- **โมดูลการรักษาความปลอดภัย (Security Module):** ดูแลระบบ CSRF Protection, Session Management, Input Sanitization, Rate Limiting, Security Headers และ Activity Logging
- **โมดูลผู้ดูแลระบบ (Administration Module):** ระบบหลังบ้านสำหรับจัดการสมาชิก บรรณานุกรม โครงการ ประกาศ ข้อเสนอแนะ บันทึกกิจกรรม การตั้งค่าระบบ และการสำรองข้อมูล
- **โมดูลอีเมล (Email Module):** ใช้ PHPMailer ในการส่งอีเมลผ่าน SMTP สำหรับยืนยันการสมัครสมาชิก และลิงก์รีเซ็ตรหัสผ่าน

**3) ชั้นข้อมูล (Data Layer)**

ชั้นข้อมูลทำหน้าที่จัดเก็บและจัดการข้อมูลทั้งหมดของระบบ ใช้ระบบจัดการฐานข้อมูลเชิงสัมพันธ์ MySQL เวอร์ชัน 8.0 รองรับชุดอักขระ UTF-8MB4 (utf8mb4_unicode_ci) เพื่อรองรับภาษาไทยและอีโมจิได้อย่างถูกต้อง การเชื่อมต่อฐานข้อมูลทั้งระบบใช้ PDO (PHP Data Objects) พร้อมกำหนดค่า Prepared Statements เป็นค่าตั้งต้น เพื่อป้องกันการโจมตี SQL Injection ค่าการตั้งต่อฐานข้อมูลจัดเก็บในไฟล์ `.env` เพื่อแยกข้อมูลที่มีความอ่อนไหว (Sensitive Data) ออกจากซอร์สโค้ด โดยมี Environment Loader ทำหน้าที่โหลดค่าตัวแปรสภาพแวดล้อมเข้าสู่ระบบ นอกจากนี้ยังมีระบบ File-based Cache สำหรับเก็บผลลัพธ์การค้นหา Smart Search เป็นเวลา 5 นาที เพื่อลดภาระการเรียก API ภายนอกซ้ำ

**4) ชั้นเชื่อมต่อภายนอก (External Integration Layer)**

ระบบ Babybib เชื่อมต่อกับฐานข้อมูลและบริการภายนอก 7 แหล่ง เพื่อดึงข้อมูลหนังสือ บทความวิชาการ และวิทยานิพนธ์ มาใช้ในการเติมข้อมูลแบบฟอร์มอัตโนมัติ (Auto-fill) ดังแสดงในตารางที่ 4.1

**ตารางที่ 4.1** แหล่งข้อมูลภายนอกที่ระบบเชื่อมต่อ

| ลำดับ | แหล่งข้อมูล | ประเภท API | ข้อมูลที่ได้รับ | ขอบเขตภาษา |
|:---:|:---|:---:|:---|:---:|
| 1 | Google Books API | REST API | ชื่อหนังสือ, ผู้แต่ง, สำนักพิมพ์, ปีที่พิมพ์, ISBN, ภาพปก | ไทย/อังกฤษ |
| 2 | Open Library API | REST API | ชื่อหนังสือ, ผู้แต่ง, สำนักพิมพ์, จำนวนหน้า, ครั้งที่พิมพ์ | สากล |
| 3 | CrossRef API | REST API | บทความวารสาร, DOI, ชื่อวารสาร, ปีที่, ฉบับที่ | สากล |
| 4 | OpenAlex API | REST API | บทความวิชาการ, DOI, สถาบัน, ผู้แต่ง | สากล |
| 5 | Semantic Scholar API | REST API | บทความวิชาการ, DOI, จำนวนการอ้างอิง | สากล |
| 6 | ThaiLIS (TDC) | Web Scraping | วิทยานิพนธ์ไทย, ชื่อเรื่อง, ผู้แต่ง, สถาบัน | ไทย |
| 7 | Library of Congress | REST API | ข้อมูลบรรณานุกรม, ผู้แต่ง, ปีพิมพ์ | สากล |

ภาพรวมสถาปัตยกรรมระบบทั้ง 4 ชั้น แสดงดังภาพที่ 4.1

### 4.1.2 การออกแบบฐานข้อมูล (Database Design)

ระบบ Babybib ใช้ระบบจัดการฐานข้อมูลเชิงสัมพันธ์ MySQL ในการจัดเก็บข้อมูล โดยฐานข้อมูลมีชื่อว่า `babybib_db` ใช้ชุดอักขระ UTF-8MB4 (utf8mb4_unicode_ci) เพื่อรองรับอักขระภาษาไทย ภาษาอังกฤษ อักขระพิเศษ และอีโมจิได้อย่างสมบูรณ์ ฐานข้อมูลประกอบด้วยตารางทั้งหมด 11 ตาราง ดังแสดงในตารางที่ 4.2

**ตารางที่ 4.2** รายละเอียดตารางในฐานข้อมูล babybib_db

| ลำดับ | ชื่อตาราง | จำนวนฟิลด์ | คำอธิบาย |
|:---:|:---|:---:|:---|
| 1 | users | 21 | จัดเก็บข้อมูลสมาชิก ได้แก่ ชื่อผู้ใช้ ชื่อ-นามสกุล อีเมล รหัสผ่าน (เข้ารหัส bcrypt) ประเภทองค์กร จังหวัด สิทธิ์การใช้งาน (user/admin) สถานะนักศึกษา LIS/CMU รูปประจำตัว ภาษาที่ใช้งาน และสถิติการใช้งาน |
| 2 | email_verifications | 7 | จัดเก็บรหัสยืนยันอีเมลสำหรับการสมัครสมาชิก ประกอบด้วย รหัส OTP 6 หลัก วันหมดอายุ และสถานะการยืนยัน |
| 3 | password_resets | 7 | จัดเก็บ Token สำหรับรีเซ็ตรหัสผ่านผ่านลิงก์อีเมล (มีอายุ 15 นาที) |
| 4 | resource_types | 10 | กำหนดประเภททรัพยากรอ้างอิงที่ระบบรองรับ จำนวน 38 ประเภท จัดเป็น 10 หมวดหมู่ โดยจัดเก็บการตั้งค่าฟิลด์ที่ต้องแสดงในรูปแบบ JSON (fields_config) |
| 5 | projects | 7 | จัดเก็บโครงการวิจัยของผู้ใช้ เพื่อจัดกลุ่มบรรณานุกรม โดยผู้ใช้แต่ละคนสร้างได้สูงสุด 30 โครงการ |
| 6 | bibliographies | 13 | ตารางหลักสำหรับจัดเก็บบรรณานุกรมที่สร้าง เก็บข้อมูลดิบในรูปแบบ JSON ข้อความบรรณานุกรมที่จัดรูปแบบแล้ว การอ้างอิง 2 รูปแบบ ภาษา คีย์สำหรับจัดเรียง ปีพิมพ์ และลำดับอักษร ผู้ใช้แต่ละคนสร้างได้สูงสุด 300 รายการ |
| 7 | activity_logs | 9 | บันทึกกิจกรรมการใช้งานระบบ ได้แก่ การกระทำ คำอธิบาย IP Address User Agent และเวลา (ลบอัตโนมัติหลัง 7 วันสำหรับสมาชิกทั่วไป เก็บถาวรสำหรับ Admin) |
| 8 | announcements | 10 | จัดเก็บประกาศจากผู้ดูแลระบบ รองรับ 2 ภาษา (ไทย/อังกฤษ) มีกำหนดวันเริ่มต้นและสิ้นสุดการแสดงผล |
| 9 | feedback | 10 | จัดเก็บข้อเสนอแนะจากผู้ใช้ มีสถานะ 3 ระดับ (รอดำเนินการ, อ่านแล้ว, แก้ไขแล้ว) พร้อมช่องตอบกลับจากผู้ดูแลระบบ |
| 10 | system_settings | 4 | จัดเก็บการตั้งค่าระบบส่วนกลาง เช่น ชื่อเว็บไซต์ จำนวนบรรณานุกรมสูงสุดต่อผู้ใช้ สถานะ Maintenance Mode ภาษาตั้งต้น รายชื่อจังหวัดของประเทศไทย 77 จังหวัด |
| 11 | page_visits | 6 | จัดเก็บสถิติการเข้าชมเว็บไซต์รายวัน แยกตามจำนวนครั้งรวมและจำนวนผู้เยี่ยมชมที่ไม่ซ้ำ (Unique Visitors) ไม่นับรวมบอท (Bot) |

ความสัมพันธ์ระหว่างตาราง (Entity Relationships) ที่สำคัญมีดังนี้

- ตาราง `bibliographies` มีความสัมพันธ์แบบ Many-to-One กับตาราง `users`, `resource_types` และ `projects` หมายความว่าบรรณานุกรมหนึ่งรายการเป็นของผู้ใช้คนเดียว มีประเภททรัพยากรหนึ่งประเภท และอยู่ในโครงการเดียว (หรือไม่มี)
- ตาราง `projects` มีความสัมพันธ์แบบ Many-to-One กับตาราง `users` หมายความว่าโครงการหนึ่งเป็นของผู้ใช้คนเดียว
- ตาราง `email_verifications` และ `password_resets` มีความสัมพันธ์แบบ Many-to-One กับตาราง `users` โดยกำหนด ON DELETE CASCADE เมื่อลบบัญชีผู้ใช้ ข้อมูลในตารางเหล่านี้จะถูกลบตามโดยอัตโนมัติ
- ตาราง `activity_logs`, `feedback` และ `announcements` มีความสัมพันธ์กับตาราง `users` โดยกำหนด ON DELETE SET NULL เมื่อลบบัญชีผู้ใช้ ข้อมูลในตารางเหล่านี้จะยังคงอยู่แต่ค่า user_id จะกลายเป็น NULL

### 4.1.5 ระบบรักษาความปลอดภัย (Security Implementation)

ระบบ Babybib ได้รับการออกแบบให้มีมาตรการรักษาความปลอดภัยตามมาตรฐานสากล ครอบคลุมทั้งด้านการป้องกันการโจมตี การเข้ารหัสข้อมูล และการควบคุมการเข้าถึง ดังแสดงในตารางที่ 4.7

**ตารางที่ 4.7** มาตรการรักษาความปลอดภัยของระบบ

| ลำดับ | มาตรการ | รายละเอียดการดำเนินการ |
|:---:|:---|:---|
| 1 | **ป้องกัน SQL Injection** | ใช้ PDO Prepared Statements พร้อมตั้งค่า `PDO::ATTR_EMULATE_PREPARES = false` ในการเชื่อมต่อฐานข้อมูลทุกจุดของระบบ ไม่มีการต่อสตริง SQL โดยตรงจากข้อมูลที่ผู้ใช้ป้อน |
| 2 | **ป้องกัน Cross-Site Request Forgery (CSRF)** | สร้าง CSRF Token ด้วย `bin2hex(random_bytes(32))` จัดเก็บใน Session และตรวจสอบทุก POST Request ด้วยฟังก์ชัน `hash_equals()` เพื่อป้องกัน Timing Attack |
| 3 | **ป้องกัน Cross-Site Scripting (XSS)** | ใช้ `htmlspecialchars()` กับ ENT_QUOTES และ UTF-8 ในทุกจุดที่แสดงผลข้อมูลจากผู้ใช้ กำหนด Content Security Policy (CSP) Header เพื่อจำกัดแหล่งที่มาของสคริปต์ |
| 4 | **การเข้ารหัสรหัสผ่าน** | ใช้อัลกอริทึม bcrypt ผ่านฟังก์ชัน `password_hash()` ของ PHP ในการเข้ารหัสรหัสผ่าน และ `password_verify()` ในการตรวจสอบ มีการกำหนดความแข็งแรงของรหัสผ่านขั้นต่ำ 8 ตัวอักษร ต้องมีตัวพิมพ์ใหญ่ ตัวพิมพ์เล็ก และตัวเลข |
| 5 | **การจัดการ Session** | ใช้ `session_regenerate_id(true)` หลังล็อกอินเพื่อป้องกัน Session Fixation กำหนด Cookie เป็น HTTPOnly, SameSite=Lax กำหนด Session Timeout 10 นาทีเมื่อไม่มีกิจกรรม ระบบจะลบ Session และบังคับล็อกเอาท์อัตโนมัติ |
| 6 | **HTTP Security Headers** | กำหนด 7 Security Headers ได้แก่ X-Content-Type-Options: nosniff, X-Frame-Options: SAMEORIGIN, X-XSS-Protection: 1; mode=block, Referrer-Policy: strict-origin-when-cross-origin, Permissions-Policy (ปิด geolocation/microphone/camera), Content-Security-Policy, Strict-Transport-Security (HSTS) |
| 7 | **Rate Limiting** | จำกัดจำนวน Request ของ Smart Search API ที่ 30 ครั้งต่อนาทีต่อ IP Address ใช้ File-based tracking ด้วยการ Hash IP Address หากเกินกำหนดจะส่ง HTTP 429 Too Many Requests |
| 8 | **Input Validation & Sanitization** | มีชุดฟังก์ชัน Validation สำหรับทุกประเภทข้อมูล ได้แก่ validateEmail(), validatePassword(), validateUsername(), validateInt(), validateString(), validateDate(), validateFileUpload() โดย validateFileUpload() มีการตรวจจับโค้ด PHP ที่ซ่อนอยู่ในไฟล์รูปภาพ |
| 9 | **บันทึกกิจกรรม (Activity Logging)** | บันทึกทุกกิจกรรมสำคัญลง activity_logs พร้อม IP Address และ User Agent เพื่อใช้ตรวจสอบย้อนหลัง |
| 10 | **Security Event Logging** | บันทึกเหตุการณ์ด้านความปลอดภัยลงไฟล์ `logs/security.log` ในรูปแบบ JSON ประกอบด้วย Timestamp, Event, Severity, IP, User Agent และรายละเอียด |
| 11 | **การแยกข้อมูลอ่อนไหว** | จัดเก็บค่าตั้ง Database, API Keys, SMTP Credentials ไว้ในไฟล์ `.env` ที่อยู่ใน `.gitignore` ไม่ถูกอัปโหลดขึ้น Version Control |
